Execution Spine

One flow, four surfaces

Surface Role Primary concern
Dashboard React UI tại `:3000` cho leaderboard, audit view, settings và ops tools. Human-facing review experience.
API Server FastAPI tại `:8000` giữ routers, orchestration và engine entrypoints. Stable contracts and task coordination.
Persistence PostgreSQL lưu audit history, overrides, engine settings và telemetry. Operational memory over time.
Documentation MkDocs tại `:8001` chứa ADR, API specs, feature notes và rules. Institutional knowledge, not runtime state.
Deployment Notes
  • `./manage.sh` boot toàn bộ stack theo Docker Compose, giúp dev environment bám sát production contract.
  • Nginx proxy ` /api/* ` về backend, nên dashboard giữ URL shape ổn định mà không lộ topology ra ngoài.
  • SSE được dùng cho realtime logs thay vì polling nặng tay trong lúc audit chạy.
Review Lens

Slide này không đi sâu vào feature cụ thể. Điều cần khóa ở đây là ai sở hữu state nào, hệ nào chịu trách nhiệm hiển thị, và tầng nào chỉ nên đóng vai trò documentation thay vì runtime source of truth.

Topology matters because UI, API, DB and Docs each solve a different failure mode. No single layer should silently own everything.

5-Step Audit Pipeline

A deliberate progression, not one giant AI call

01
Discovery
Detect source directories, classify files by module and compute effective LOC.
02
Static Scan
AST + Regex catch deterministic issues before any AI budget is consumed.
03
AI Gatekeeper
Context-aware filter to remove false positives and reduce noisy ledger rows.
04
Aggregation
Penalty rolls into pillars, K-factor normalization and weighted project score.
05
Reporting
Dashboard, history, member ranking and streaming logs expose the result surface.
Architectural Intent
  • Đắt tiền nhất trong pipeline không được chạy trước. AI phải vào sau khi hệ tĩnh đã rút gọn bề mặt nghi ngờ.
  • Mỗi bước có output rõ ràng, nên việc điều tra regression hoặc cost spike không bị mù.
Reviewer Questions
  • Stage boundaries có đủ rõ để thêm feature mới mà không làm pipeline monolithic hơn không?
  • Realtime experience hiện tại có phản ánh đúng tiến độ backend, hay chỉ là UI animation đẹp mắt?
Static scanner sets the floor; AI refines, not replaces. That separation is the cost-control mechanism.

Pillar Model

Penalty becomes a business-readable score

Security
K = 0.5
Cực nhạy với lỗi nghiêm trọng, phù hợp cho secrets hoặc security violations.
Performance
K = 2.0
Dùng cho inefficiency patterns, hot loops và vấn đề query hiệu năng.
Reliability
K = 2.0
Tập trung vào error handling, crash risk và unstable flows.
Maintainability
K = 4.0
Chấp nhận dung sai lớn hơn nhưng vẫn phản ánh debt tích lũy.
pillar_score = 10 / (1 + penalty / K_FACTOR)
Normalization Rules
  • Penalty được chuẩn hóa theo effective LOC / 1000 để project lớn không bị đối xử như project nhỏ.
  • Final score dùng weighted average theo LOC của module, nên phần code lớn có ảnh hưởng tương xứng.
  • Member scoring dựa trên git blame trong cửa sổ thời gian cấu hình, tránh đánh đồng toàn đội.
Why Tech Leads Care

Hệ điểm chỉ hữu ích nếu nó so sánh được giữa các project có quy mô khác nhau. Nếu normalization yếu, dashboard sẽ biến thành bảng trang trí thay vì công cụ ra quyết định.

Scoring is not just math; it is the contract between findings and action. Normalization protects that contract.
State Boundaries Persistence versus fallback
05 / 06
Config / Fallback

In-memory is a convenience, not durable state

Khi thiếu `DATABASE_URL`, app vẫn có thể boot repository list từ `config.py`, nhưng đó là fallback để dev flow sống tiếp chứ không phải đường vận hành đầy đủ.

Persistent State

History, overrides and settings need Postgres

Audit history, rule overrides, engine settings và các lớp state vận hành chỉ có ý nghĩa nếu dữ liệu được giữ xuyên phiên làm việc.

Documentation Layer

MkDocs is institutional memory

Tài liệu phải giúp team hiểu hệ thống, nhưng không được ngầm trở thành nơi lưu runtime truth thay cho DB hoặc API.

Boundary Checklist
  • UI có nói rõ chỗ nào chỉ là snapshot, chỗ nào là persisted record chưa?
  • Fallback no-DB có đủ rõ để tránh người dùng tưởng history vẫn đang được lưu không?
config.py -> repository bootstrap
postgres -> audit history / rules / settings
mkdocs -> contracts / ADR / feature knowledge
Confusing fallback with persistence creates operational debt fast. This boundary must stay explicit in both UI and docs.
